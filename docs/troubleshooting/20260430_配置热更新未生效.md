# 故障复盘报告
- **日期**: 2026-04-30
- **严重级别**: Medium

## 1. 问题现象
运行中的 API / Telegram Bot 在用户修改 `.env` 或 `config/config.yaml` 后，处理结果仍继续沿用旧配置：

- 删除 `TELEGRAM_NOTIFY_CHAT_ID` 后，Telegram 成功结果推送仍可能继续使用旧 chat_id。
- 修改 `summary_upload.*.enabled` 后，不重启 API 进程时上传行为仍按旧配置执行。

## 2. 根本原因 (Root Cause)
- `utils.summary_uploader.load_env()` 之前仅调用 `load_dotenv()` 叠加环境变量。因为 `load_dotenv()` 不会清理已从 `.env` 删除的键，所以长驻进程会继续保留历史值。
- `utils.api_queue.QueueManager._ensure_config()` 之前只在首次请求时加载一次 `config/config.yaml`，后续请求直接复用缓存配置，没有基于文件变更时间刷新。

## 3. 解决方案
- 重写 `load_env()` 的加载策略：每次读取 `.env` 文件后同步到 `os.environ`，并清理已从 `.env` 删除的旧键，确保环境变量删除/修改能立即生效。
- 为 `QueueManager` 增加配置文件路径与 `mtime_ns` 检测；当 `config/config.yaml` 发生变化时自动重新加载配置。
- 新增回归测试覆盖：
  - `.env` 删除键后不再保留旧值
  - `QueueManager` 在配置文件修改后自动读取新配置

## 4. 预防措施
- 对长驻进程读取的配置统一采用“显式同步 + 变更检测”策略，避免只在启动阶段加载一次。
- 将 `.env` 热更新与 `config.yaml` 热更新加入回归测试，防止后续重构回退为静态缓存行为。
