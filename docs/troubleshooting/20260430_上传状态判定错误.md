# 故障复盘报告
- **日期**: 2026-04-30
- **严重级别**: Medium

## 1. 问题现象
在命令传入 `--id` 且 `config/config.yaml` 中 `summary_upload.lis_rss.enabled=false` 的情况下，流程日志仍可能出现 “LIS-RSS: ✅ 成功” 的汇总结果，与配置语义冲突。

同时，上传结果里即使只有 `Blinko` 成功，主流程仍可能误判为“所有上传任务均失败”。

## 2. 根本原因 (Root Cause)
上传编排层把“是否传入文章 ID”和“子系统是否在配置中启用”拆成了两套判断：

- `main.py` 只根据是否提供 `id` 决定 `skip_lis_rss`
- `utils/summary_uploader.py` 内部又根据 `summary_upload.lis_rss.enabled` 判断是否禁用

由于 `upload_to_lis_rss()` 在配置禁用时直接返回 `True`，所以虽然实际没有执行上传，但汇总阶段会把它当成成功显示。

此外，`main.py::_is_all_upload_failed()` 在统计成功任务时遗漏了 `blinko`，导致“只有 Blinko 成功”的场景仍会被误判为全部失败。

## 3. 解决方案
本次修复将上传阶段统一为“成功 / 失败 / 跳过”三态语义：

- 在 `utils/summary_uploader.py` 的 `upload_all()` 中集中判断子系统是否应执行
- 对“配置禁用”“未提供文章ID”“本次流程未启用”等情况统一记录到 `_skipped` 和 `_skip_reasons`
- 汇总日志改为显示明确的跳过原因，不再把跳过伪装成成功
- 修复 `main.py::_is_all_upload_failed()`，将 `blinko` 纳入统计，并避免“全部跳过”被误判为失败
- 修复 `scripts/upload_only.py` 的汇总与退出码判断，使其忽略被跳过的子系统

## 4. 预防措施
- 新增 `tests/test_summary_uploader.py`
- 覆盖 `lis_rss.enabled=false` 时应标记为跳过而非成功
- 覆盖仅 `Blinko` 成功时不应判定为“全部上传失败”
- 覆盖全部子系统均跳过时不应误判为失败
